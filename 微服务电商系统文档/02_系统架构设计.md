# 02 系统架构设计

## 1. 架构风格与拆分原则

采用 **DDD 限界上下文**思路拆分，每个微服务满足：
- **单一职责**：一个服务对应一个业务能力（用户、商品、库存…）。
- **独立数据库**（database-per-service）：服务间**不允许直接连别人的库**，只能走 API/MQ。这是
  面试高频考点，务必坚守，否则就是"分布式单体"。
- **无状态**：会话用 JWT，状态进 Redis/DB，便于水平扩容与 K8s 部署。

## 2. 服务清单与端口约定

| 服务 | 端口 | 数据库 | 职责 |
|---|---|---|---|
| gateway | 9000 | — | 路由、鉴权、限流 |
| user-service | 9001 | mall_user | 账户、地址、会员 |
| product-service | 9002 | mall_product | 分类、SPU/SKU、上下架 |
| inventory-service | 9003 | mall_inventory | 库存、预占、流水 |
| cart-service | 9004 | (Redis) | 购物车 |
| order-service | 9005 | mall_order | 订单、状态机 |
| payment-service | 9006 | mall_payment | 支付、回调、对账 |
| search-service | 9007 | (ES) | 商品检索 |
| seckill-service | 9008 | mall_seckill + Redis | 秒杀 |

基础设施默认端口：Nacos 8848、Sentinel Dashboard 8858、Seata 8091、RocketMQ NameSrv 9876、
Redis 6379、MySQL 3306、ES 9200、SkyWalking OAP 11800/UI 8080、Prometheus 9090、Grafana 3000。

## 3. 关键调用链（下单为例）

```
Client ──▶ Gateway ──(JWT校验)──▶ order-service.createOrder
   order-service 开启 Seata 全局事务 (@GlobalTransactional):
     1) Feign 调 product-service 查价/校验上下架
     2) Feign 调 inventory-service 预占库存   ← 分支事务
     3) Feign 调 promotion(或order内) 算券    ← 分支事务
     4) 本地插入订单 (待支付)                   ← 分支事务
   提交全局事务 → 返回订单号
   发送 RocketMQ 延时消息(30min) 用于超时取消
Client ──▶ payment-service.pay ──▶ 支付沙箱
   回调 ──▶ payment 验签+幂等 ──▶ 发 MQ "支付成功"
     order-service 消费 → 订单转"已支付" + 通知 inventory 预占转实扣
```

秒杀链路**不走 Seata**（性能），改最终一致，见 `05`。

## 4. 服务间通信

- **同步**：OpenFeign（声明式）+ Spring Cloud LoadBalancer + Sentinel 熔断降级。Feign 接口
  放在各服务的 `xxx-api` 子模块，供调用方依赖（只含 DTO 和接口，不含实现）。
- **异步**：RocketMQ。用于：订单超时取消（延时消息）、支付结果通知、库存变更同步 ES、秒杀异步落库。

## 5. 工程结构（Maven 多模块）

```
beibu-mall/                      父 POM，统一依赖与版本管理(dependencyManagement)
├── mall-common/                 通用：统一响应Result、异常、枚举、工具、常量
├── mall-gateway/
├── mall-user/
│   ├── mall-user-api/           对外Feign接口 + DTO（被别的服务依赖）
│   └── mall-user-server/        实现：controller/service/mapper/entity
├── mall-product/  (api + server)
├── mall-inventory/(api + server)
├── mall-cart/
├── mall-order/    (api + server)
├── mall-payment/  (api + server)
├── mall-search/
├── mall-seckill/  (api + server)
└── docker/                      各中间件的 compose 与配置
```

**单服务内部分层**（每个 `*-server`）：
```
controller   → 入参校验、调 service、返回 Result
service       → 业务编排、事务、调 mapper/feign/mq/redis
  └ impl
mapper        → MyBatis-Plus，仅访问本服务库
entity / po   → 数据库实体
dto / vo      → 传输/展示对象（不要把 entity 直接暴露给前端）
config        → 各类配置类
feign         → 对外部服务的 Feign 客户端
mq            → 生产者/消费者
```

## 6. 统一规范（所有服务遵守）

- **统一响应**：`Result<T>{ code, msg, data }`，放在 `mall-common`。
- **统一异常**：自定义 `BizException(code, msg)` + 全局 `@RestControllerAdvice`。
- **统一日志**：logback 输出 JSON 或固定格式，含 traceId（SkyWalking 注入）。
- **参数校验**：`jakarta.validation` 注解 + 全局校验异常处理。
- **幂等**：写接口（支付回调、下单、MQ 消费）必须幂等（唯一索引 / Redis token / 去重表）。
- **配置外置**：所有配置进 Nacos 配置中心，按 `服务名-环境.yaml` 管理（dev/test/prod）。

## 7. 配置中心接入要点（Nacos 3.x）

> 新版 Nacos/SCA 已**移除 bootstrap 方式**，改用 `spring.config.import`。务必告知 AI，否则会用过时写法。

```yaml
# application.yml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      server-addr: ${NACOS_ADDR:127.0.0.1:8848}
      discovery: { namespace: ${ENV:dev} }
      config:   { namespace: ${ENV:dev}, file-extension: yaml }
  config:
    import:
      - optional:nacos:${spring.application.name}.yaml
      - optional:nacos:common-shared.yaml   # 公共配置
```

## 8. 环境分层

- **本地开发**：Docker Compose 起全部中间件 + IDEA 跑各服务（或部分服务）。
- **集成测试**：Testcontainers 临时拉起 MySQL/Redis/RocketMQ（见 06）。
- **演示/生产演练**：K8s（k3s/minikube 起步），见 07。
