# 10 全套提示词手册（复制即用）

## 怎么用（记住这 3 步）

1. **开新对话** → 先把【第 0 步：开场设定】整段发给 AI，等它回"明白"。
2. **发当周任务** → 从下面按周复制对应提示词发出去。
3. **拿到代码后** → 不懂就用【通用追问】里的话继续问，直到你能复述。

> 同一个对话可以连着做好几周；如果对话太长卡顿了，就开新对话、重新发一次【开场设定】再继续。

## 术语速查（看不懂就回这里）

| 词 | 一句话解释 |
|---|---|
| 微服务 | 把一个大系统拆成多个能各自独立运行、独立部署的小程序 |
| Maven 多模块 / 父 POM | 一个总项目下挂多个子项目；父 POM 是统一管版本和依赖的"总配置" |
| dependencyManagement | 父 POM 里统一规定各依赖用哪个版本，子项目跟着用，不用各写各的 |
| 注册中心 / Nacos | 服务的"通讯录"，服务启动后在这里登记，别人才找得到它（Nacos 还兼配置中心） |
| 配置中心 | 把各服务的配置集中放一处管理，改配置不用重启改代码 |
| OpenFeign / Feign | 让 A 服务"像调用本地方法一样"去调用 B 服务的接口 |
| RocketMQ / MQ / 消息队列 | 服务间异步通信的"快递站"，A 发消息进去，B 慢慢取，互不等待 |
| 网关 / Gateway | 所有请求的统一大门，负责转发、登录校验、限流 |
| JWT | 登录后发给你的一张"令牌"，之后带着它就证明你已登录 |
| Seata / 分布式事务 / 全局事务 | 跨多个服务的操作要么全成功要么全回滚的机制 |
| Sentinel / 限流 / 熔断 / 降级 | 流量保护：限流=控制进来的请求数；熔断=下游坏了就先断开；降级=坏了给个兜底结果 |
| 乐观锁 | 改数据时带上"版本号"判断有没有被别人改过，用来防止超卖 |
| 预占库存 | 下单先把库存"锁住"，付款后才真扣，超时没付就释放 |
| Redis | 超快的内存数据库，常用来做缓存、计数、分布式锁 |
| Redisson / 分布式锁 | 用 Redis 实现的"锁"，保证同一时刻只有一个请求能操作某资源 |
| Lua 脚本 | 在 Redis 里一次性原子执行的一小段逻辑，秒杀扣库存用它防超卖 |
| 缓存穿透/击穿/雪崩 | 缓存的三种翻车情况（查不存在的、热点过期、大量同时过期），各有应对 |
| 幂等 | 同一个请求重复发多次，结果和发一次一样（防重复下单/重复扣款） |
| SPU / SKU | SPU=一种商品(对虾)，SKU=具体规格(鲜活500g装) |
| DTO / VO / entity | entity=数据库表对应的对象；DTO/VO=传给前端用的对象（不直接暴露表结构） |
| 单元测试 | 只测一个方法的逻辑，把外部依赖"假装"掉，跑得飞快 |
| 集成测试 | 连真数据库/Redis 一起测，更接近真实 |
| Testcontainers | 测试时用 Docker 临时拉起一个真 MySQL/Redis，测完即删 |
| 契约测试 | 保证 A 调 B 的接口，B 改了不会偷偷把 A 弄崩 |
| JaCoCo / 覆盖率 | 统计你的测试覆盖了多少比例代码，低于设定值就不让通过 |
| CI/CD | 每次提交代码自动跑测试、自动构建，省去手动操作 |
| Docker / 镜像 / 容器 | 把程序连环境打成一个"镜像"，到哪都能一键跑成"容器" |
| Docker Compose | 用一个文件一键启动多个容器（如一次起 MySQL+Redis+Nacos） |
| Kubernetes / K8s | 容器的"管理调度系统"，负责部署、扩容、自愈 |
| Prometheus / Grafana / SkyWalking | 监控指标 / 画监控图表 / 看请求在各服务间怎么流转 |

## 第 0 步：开场设定（每次新对话先发这段）⭐

```
你是一名资深 Java 后端微服务工程师，带我从零做一个练手用的微服务电商项目「北部湾海鲜商城」。
我是大学生，做过单体 Spring Boot 项目，但没做过微服务，很多术语不熟。

请全程遵守以下设定：

【技术栈版本】务必用这些版本和写法，不要用更老的：
- JDK 21
- Spring Boot 3.5.x
- Spring Cloud 2025.0.x
- Spring Cloud Alibaba 2025.0.x
- MyBatis-Plus、MySQL 8、Redis 7
- Nacos 3.x（接配置中心请用 spring.config.import 写法，不要用过时的 bootstrap.yml）

【架构铁律】
- 每个业务做成一个独立服务，每个服务有自己独立的数据库；
- 服务之间只能通过接口调用(OpenFeign)或消息队列(RocketMQ)通信；
- 绝对不允许一个服务直接连另一个服务的数据库。

【代码风格】
- 分 controller / service / mapper 三层；
- 接口返回统一用一个 Result 包装类，出错抛自定义 BizException；
- 不要把数据库实体直接返回给前端，用 DTO/VO。

【教学要求】很重要
- 你每给一段代码，先用大白话讲清"这段在做什么、为什么这么写"；
- 假设我术语基础差，凡是出现专业术语就顺手用一句话解释；
- 一次只做一个小功能，不要一次性把整个项目都生成出来。

明白请回复"明白，我们开始吧"，然后等我发第一个任务。
```

```
帮我把各个服务打包成 Docker 镜像。请先解释 Docker、镜像、容器分别是什么。然后：
1. 给每个服务写一个 Dockerfile（用多阶段构建，解释为什么这样镜像更小）；
2. 写一个 docker-compose 文件，能一键把"所有中间件 + 所有业务服务"都启动起来，方便演示；
3. 一步步教我怎么构建镜像、怎么一键启动、怎么验证整套系统跑起来了。
```

**提示词 12.2（K8s，量力而行）**
```
我想把核心的 2~3 个服务部署到 Kubernetes 练手。请先用大白话讲清 K8s 是干嘛的、和 Docker 什么关系，
并推荐一个本机就能跑的轻量集群（如 k3s 或 minikube），给安装步骤。然后：
为订单服务写 Deployment 和 Service 的 yaml（多副本、配好健康检查探针），用 Ingress 暴露网关。
逐步教我怎么把镜像部署上去、怎么访问、怎么查看 Pod 状态。解释每个 yaml 字段的含义。
```

**提示词 12.3（监控可观测）**
```
给项目加监控。请逐一解释 Prometheus、Grafana、SkyWalking 各是做什么的。然后教我：
1. 让各服务暴露监控指标，用 Prometheus 采集、Grafana 画出 QPS 和响应时间的看板；
2. 接入 SkyWalking 看一个请求在多个服务之间是怎么流转的（调用链路）。
给出配置和 docker 启动方式，并告诉我哪些界面截图适合放进项目 README 和简历。
```

## 通用提示词（任何时候都能用）

**① 没看懂代码 / 要它再讲**
```
上面这段代码我没完全看懂，请用更通俗的话、配一个生活中的例子，重新讲一遍它在做什么、
为什么需要它。假设我是完全的新手。
```

**② 报错了 / 排查**
```
我运行后报错了，错误信息如下：
【把完整报错粘进来】
相关代码：
【把相关代码粘进来】
请帮我找出根本原因，给出最小的修改方案，并解释为什么会这样错。不要重写整个文件。
```

**③ 验收追问（确认自己真懂了，为面试准备）**
```
针对刚才实现的这个功能，请出 3 道面试官可能会问的问题，并给出参考答案。
然后假设我答得不好，追问我一个更深的问题，帮我查漏补缺。
```

**④ 想偷懒时提醒自己**
```
这个功能请你不要直接全写完，先讲思路和方案，我理解确认后你再分步给代码——
我要的是学会，不是抄完。
```

## 一句话提醒

提示词只是脚手架。**真正决定简历含金量的，是你能不能把每个功能用自己的话讲给别人听。**
所以每做完一块，别急着进下一周——先用【通用提示词③】考考自己。
