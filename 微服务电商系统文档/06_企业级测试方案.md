# 06 企业级测试方案 ⭐

> 这是你"工程素养"的证明。很多学生项目"没有测试"，有一套像样的测试体系是巨大加分项。
> 核心思想：**测试金字塔**——大量快而便宜的单元测试在底，少量慢而贵的端到端在顶。

```
        ▲  少
      E2E / 系统测试        (Postman/Newman, 真起服务, 慢)
     契约测试 (Contract)     (服务间接口兼容, Spring Cloud Contract)
   集成测试 (Integration)    (Testcontainers 起真 MySQL/Redis/MQ)
  单元测试 (Unit)            (JUnit5 + Mockito + AssertJ, 海量, 快)
        ▼  多
   旁路：性能/压力测试(JMeter)、混沌测试(可选, ChaosBlade)
```

---

## 1. 工具选型

| 层 | 工具 | 说明 |
|---|---|---|
| 单元 | JUnit 5 + Mockito + AssertJ | mock 掉外部依赖，只测本类逻辑 |
| 集成 | Spring Boot Test + **Testcontainers** | Docker 临时拉真中间件，比 H2/嵌入式更真实 |
| 契约 | Spring Cloud Contract（或 Pact） | 保证 Feign 调用方/被调方接口不破坏 |
| 接口/E2E | Apifox / Postman + **Newman**(CI跑) | 黑盒接口回归 |
| 覆盖率 | **JaCoCo** + CI 门禁 | 行覆盖率 < 阈值则构建失败 |
| 性能 | **JMeter**（或 Gatling/k6） | 秒杀、下单压测 |
| 混沌(选) | ChaosBlade | 注入延迟/宕机验证降级 |

---

## 2. 单元测试（地基，写最多）

**原则**：测 Service 的业务分支、边界、异常路径；用 Mockito mock 掉 Mapper/Feign/Redis/MQ。
不连任何外部资源，毫秒级，能跑成百上千个。

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {
    @Mock InventoryFeignClient inventoryClient;
    @Mock OrderMapper orderMapper;
    @InjectMocks OrderServiceImpl orderService;

    @Test
    void should_throw_when_stock_not_enough() {
        when(inventoryClient.lock(any())).thenReturn(Result.fail(10031, "库存不足"));
        assertThatThrownBy(() -> orderService.createOrder(dto))
            .isInstanceOf(BizException.class)
            .extracting("code").isEqualTo(10031);
        verify(orderMapper, never()).insert(any());   // 验证未落库
    }
}
```
**要测的典型分支**：库存不足、商品下架、券不可用、金额计算（按重计价的活鲜！）、状态机非法流转、幂等命中。

## 3. 集成测试（Testcontainers，最能体现"企业级"）

不用 H2 假数据库，而是用 Testcontainers 在测试时**真起一个 MySQL/Redis 容器**，跑完即销毁，
保证测试环境与生产一致（H2 的 SQL 方言差异常掩盖 bug）。

```java
@SpringBootTest
@Testcontainers
class InventoryIntegrationTest {
    @Container
    static MySQLContainer<?> mysql = new MySQLContainer<>("mysql:8.0")
        .withDatabaseName("mall_inventory");
    @Container
    static GenericContainer<?> redis = new GenericContainer<>("redis:7-alpine")
        .withExposedPorts(6379);

    @DynamicPropertySource
    static void props(DynamicPropertyRegistry r) {
        r.add("spring.datasource.url", mysql::getJdbcUrl);
        r.add("spring.data.redis.host", redis::getHost);
        r.add("spring.data.redis.port", () -> redis.getMappedPort(6379));
    }

    @Test
    void should_not_oversell_under_concurrency() throws Exception {
        // 库存设10，开50个线程并发扣减1，断言成功数==10、最终库存==0、无负库存
        // 这条测试直接证明"乐观锁防超卖"，是简历可讲的亮点测试
    }
}
```
> 并发超卖测试、Lua 脚本扣减测试、MQ 收发测试都放这一层。RocketMQ 也有对应 Testcontainers 镜像。

## 4. 契约测试（微服务特色，区别于单体）

微服务里 A 调 B，B 改了接口可能悄悄打挂 A。契约测试让**被调方(provider)产出契约**，
调用方(consumer)按契约验证，CI 里任一方破坏契约即失败。

- 用 **Spring Cloud Contract**：provider 写 contract（Groovy/YAML），自动生成 provider 测试 +
  consumer 用的 stub。
- 适合 order↔inventory、order↔product 这种关键内部调用。
- 简历话术："用契约测试保证服务间接口演进不破坏调用方，CI 阶段拦截不兼容变更。"

---

## 5. 接口 / E2E 测试

- 用 Apifox/Postman 维护接口集合：注册→登录→加购→下单→支付→查订单 全流程。
- 导出为 Newman 脚本，在 CI 里对部署好的环境跑一遍冒烟，保证主链路不挂。

---

## 6. 覆盖率门禁（JaCoCo + CI）

Maven 配 JaCoCo 插件，设置门槛，**不达标构建失败**：
```xml
<plugin>
  <groupId>org.jacoco</groupId><artifactId>jacoco-maven-plugin</artifactId>
  <executions>
    <execution><goals><goal>prepare-agent</goal></goals></execution>
    <execution><id>check</id><goals><goal>check</goal></goals>
      <configuration><rules><rule>
        <element>BUNDLE</element>
        <limits><limit><counter>LINE</counter><value>COVEREDRATIO</value>
          <minimum>0.70</minimum></limit></limits>
      </rule></rules></configuration>
    </execution>
  </executions>
</plugin>
```
**目标**：整体行覆盖率 ≥ 70%，核心服务核心类(订单/库存/秒杀的 service) ≥ 80%。
报告截图(`target/site/jacoco/index.html`)放进项目 README，简历可写"行覆盖率 73%"。

---

## 7. 性能 / 压力测试（JMeter）

针对两条链路压测并产出对比数据：

| 场景 | 指标 | 怎么做 |
|---|---|---|
| 普通下单 | P95/P99、TPS、错误率 | 阶梯加压(50→200→500并发) |
| 秒杀 | TPS、超卖数量、DB 压力 | 对比"直连DB" vs "Redis+Lua+MQ"两版 |

**产出**：JMeter 聚合报告截图 + 一句结论（"优化后秒杀 TPS 从 ~200 提升到 ~2000，超卖 0"）。
这是简历里数据感最强的一行，面试官几乎必问"这个数怎么测的"，所以**一定要真跑**。

---

## 8. 混沌测试（可选加分）

用 ChaosBlade 给某服务注入"延迟/CPU 满载/进程杀死"，验证 Sentinel 熔断降级是否生效、主链路是否还活着。
有这一块能讲"我验证过容错"，但时间紧可放到最后。

---

## 9. 测试在 CI 里的编排

GitHub Actions（详见 07）每次 PR：
```
mvn test           # 单元 + 集成(Testcontainers 需 docker，GH Actions runner 自带)
mvn jacoco:check   # 覆盖率门禁
契约测试 / Newman 冒烟（部署后）
```
任一失败则 PR 红灯、禁止合并。**"测试不过不准合并"本身就是工程素养的体现。**

---

## 10. 测试目录约定

```
src/test/java/.../service/   单元测试  XxxServiceTest
src/test/java/.../it/        集成测试  XxxIntegrationTest (Testcontainers)
src/test/resources/          测试配置、契约、测试数据
test/jmeter/                 .jmx 压测脚本
test/postman/                接口集合
```

## 面试问答速记（测试相关）

- **测试金字塔讲一下？** → 底层单元多而快、上层E2E少而慢；避免"冰淇淋筒"(全是E2E)。
- **为什么用 Testcontainers 不用 H2？** → H2 方言/行为与 MySQL 有差异会掩盖 bug；Testcontainers 用真镜像，环境一致。
- **怎么测"防超卖"？** → 集成测试里多线程并发扣库存，断言成功数==库存数、无负库存。
- **微服务接口怎么保证不互相打挂？** → 契约测试，CI 拦截不兼容变更。
- **覆盖率怎么保证？** → JaCoCo 设门禁，低于阈值构建失败，强制写测试。
