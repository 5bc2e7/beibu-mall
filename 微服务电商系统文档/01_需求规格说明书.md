# 01 需求规格说明书（PRD）

## 1. 项目背景与领域

「北部湾海鲜商城」面向 C 端消费者销售北部湾产地海鲜。海鲜电商相比普通电商有 4 个**领域特性**，
它们正是技术复杂度的天然来源，写进项目能让面试官觉得"这人是真的在解决业务问题"：

1. **商品形态多样**：活鲜（对虾、青蟹、生蚝、石斑鱼）、冰鲜、冻品、干货（虾米、海鸭蛋、沙虫干）。
   活鲜有"按只 / 按斤"两种计价，且**称重计价**会导致下单金额需后端按实际重量结算。
2. **强时令性**：很多海鲜按汛期上市，需要**预售 / 限时抢购（秒杀）**——天然引出秒杀难点。
3. **库存敏感**：活鲜数量少、易死、损耗高，**绝不能超卖**——天然引出库存超卖与分布式锁。
4. **冷链时效**：配送区域、配送时段、冷链运费——引出履约/物流的规则计算。

> 领域设定可以适度简化，但要保留上述特性，否则项目就退化成"又一个普通商城"。

## 2. 角色与用例

| 角色 | 主要用例 |
|---|---|
| 游客 | 浏览商品、搜索、查看详情 |
| 会员 | 注册登录、管理收货地址、加购物车、下单支付、查订单、取消/退款、领券、参与秒杀 |
| 运营/管理员 | 商品上下架、库存管理、秒杀场次配置、优惠券发放、订单管理 |

## 3. 功能需求（按服务归属）

### 3.1 用户服务 user-service
- 注册（手机号/邮箱）、登录（密码 + JWT，可加短信验证码 mock）
- 用户信息、会员等级（普通/银/金，影响包邮门槛——为运费规则埋点）
- 收货地址增删改查（含默认地址）

### 3.2 商品服务 product-service
- 分类树（活鲜/冰鲜/冻品/干货 → 子类）
- SPU/SKU 模型：一个 SPU=「北部湾大对虾」，SKU=「鲜活/500g装」「冰鲜/1kg装」
- 商品属性：产地、是否活鲜、计价方式（按件/按重）、保质期、起售量
- 上下架、详情、列表分页

### 3.3 库存服务 inventory-service
- 库存查询、库存扣减/回滚、**库存预占**（下单占用，支付成功转实扣，超时释放）
- 普通库存（乐观锁）与**秒杀库存（Redis 预扣）**两套机制
- 库存流水（每次变动留痕，可对账）

### 3.4 购物车服务 cart-service
- 基于 Redis 的购物车（Hash 结构：userId → {skuId: 数量}）
- 加购、改数量、删除、选中、清空、合并（登录前后合并）

### 3.5 订单服务 order-service
- 下单（生成订单 + 调库存预占 + 调营销算价，全局事务）
- 订单状态机：待支付 → 已支付 → 待发货 → 已发货 → 已完成 / 已取消 / 退款中 / 已退款
- 超时未支付自动取消（RocketMQ 延时消息）并释放库存
- 订单列表/详情、取消、申请退款

### 3.6 支付服务 payment-service
- 发起支付（接入支付宝**沙箱**，或先做 mock 支付）
- 支付回调处理（验签、幂等）、支付成功后发 MQ 通知订单服务
- 对账任务（定时拉取支付流水与订单比对）

### 3.7 搜索服务 search-service
- 商品数据同步到 ES（MQ 异步或 Canal/定时全量+增量）
- 关键词检索、分类/价格筛选、排序、高亮、同义词（"基围虾"≈"对虾"）

### 3.8 秒杀服务 seckill-service
- 秒杀场次配置（时令海鲜限时抢）
- 秒杀下单：Redis 预扣库存 + Lua 原子 + 限流 + MQ 异步落库 + 防重复下单
- 库存预热（活动开始前把库存加载进 Redis）

### 3.9 网关 gateway
- 路由、统一 JWT 鉴权（白名单放行登录/注册/商品浏览）、全局限流、跨域、日志埋点

> 履约/物流、评价、推荐、营销券（独立）等可作为**进阶/可选服务**，时间不够先合并进订单/商品。

## 4. 非功能需求（NFR）

| 维度 | 目标 | 备注 |
|---|---|---|
| 性能 | 普通下单 P95 < 300ms；秒杀下单 TPS ≥ 1500（单机压测口径） | 用 JMeter 验证并截图进简历 |
| 可用性 | 任一非核心服务挂掉不影响浏览/下单主链路 | Sentinel 降级兜底 |
| 一致性 | 下单扣库存强一致（Seata）；秒杀最终一致 | 见 05 |
| 安全 | 密码加盐哈希、JWT、网关鉴权、支付回调验签、接口幂等 | — |
| 可观测 | 全链路追踪、核心指标看板、集中日志 | 见 07 |
| 可测试 | 行覆盖率 ≥ 70%，核心服务核心类 ≥ 80% | 见 06，CI 门禁 |
| 可维护 | 统一异常、统一响应、统一日志格式、代码规范 | 见 04、09 |

## 5. 领域核心概念词汇表（统一语言）

| 术语 | 含义 |
|---|---|
| SPU | 标准商品单元，如"北部湾大对虾" |
| SKU | 库存单元，SPU 的具体规格，如"鲜活 500g" |
| 预占库存 | 下单时锁定，未支付超时释放，支付后转实扣 |
| 活鲜 | 活体海鲜，库存敏感、易损耗、常按重计价 |
| 时令预售/秒杀 | 按汛期限时限量销售 |
| 全局事务 | 跨服务的分布式事务（Seata 管理） |

## 6. 范围边界（明确不做什么）

- 不做真实资金支付（用沙箱/mock），不做真实物流对接
- 不做复杂的推荐算法（最多做"同类目热销"）
- 前端只做能演示主流程的程度，不追求 UI 精美
- 不做多租户、不做国际化
