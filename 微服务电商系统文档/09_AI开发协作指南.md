# 09 AI 开发协作指南

> 你要"让 AI 按文档开发"。但**直接说"帮我做一个微服务商城"会得到一坨能跑不能用、你也讲不清的代码**。
> 正确做法：把项目切成小任务，每次只让 AI 做一块，你来理解、验收、串联。这样既出活，你也真学会了
> （面试官一问就露馅的项目毫无意义）。

---

## 1. 黄金原则

1. **小步快跑**：一次只让 AI 做"一个服务的一个功能 + 它的测试"，不要一把梭。
2. **先要方案再要代码**：让 AI 先说思路/接口/表，你确认后再让它写实现。
3. **你必须能讲清每一行难点**：05 章的每个难点，让 AI 解释给你听，直到你能复述。
4. **测试同步写**：每个功能让 AI 连测试一起给（对应 06），不要"以后再补"。
5. **版本钉死**：每次提醒 AI 用 README 的版本矩阵，尤其 Nacos 用 `spring.config.import`（不是 bootstrap）。

---

## 2. 喂文档的顺序

第一次对话给 AI 这几份作为"全局上下文"：`README` + `02 架构` + `03 数据库` + `04 API`。
做某个难点时再追加对应章节：做秒杀给 `05`，写测试给 `06`，部署给 `07`。
不要一次把 9 份全塞进去——AI 会抓不住重点，按当前任务给相关的即可。

---

## 3. 任务级提示词模板（直接套用）

**模板 A：搭骨架**
> 基于以下技术栈[贴 README 版本矩阵]，帮我创建 Maven 多模块父工程 beibu-mall，包含
> mall-common、mall-gateway、mall-user(api+server) 三个模块。父 POM 用 dependencyManagement
> 统一管理 Spring Boot 3.5.x / Spring Cloud 2025.0.x / Spring Cloud Alibaba 2025.0.x 版本。
> mall-common 里实现统一响应 Result、BizException、全局异常处理器。先给我目录结构和各 pom.xml，
> 我确认后再写代码。

**模板 B：实现一个服务的一个功能**
> 角色：资深 Java 微服务工程师。
> 上下文：[贴 02 该服务定义 + 03 相关表 + 04 相关接口]。
> 任务：实现 inventory-service 的"乐观锁扣减库存"功能，包含 entity/mapper/service/controller。
> 约束：用 MyBatis-Plus；扣减用单条原子 UPDATE（available>=qty）；影响行数 0 抛 BizException(10031)；
> 记库存流水。
> 同时给出：① 该 service 的 JUnit5+Mockito 单元测试（覆盖成功/库存不足）；② 一个 Testcontainers
> 集成测试，开 50 线程并发扣减、断言不超卖。
> 先解释你的实现思路和为什么这样防超卖，再给代码。

**模板 C：攻克难点（要它先教你）**
> 我要实现秒杀[贴 05 第 6 节]。请先用通俗语言给我讲清整条链路为什么这么设计：为什么扣减放 Redis、

> seckill-service 的下单接口、MQ 生产/消费、以及对应测试。

**模板 D：写测试**
> 为 [贴某 service 代码] 编写测试，遵循[贴 06]：单元测试 mock 外部依赖覆盖所有分支；
> 集成测试用 Testcontainers 起真 MySQL/Redis。目标行覆盖率该类 ≥80%。同时给 JaCoCo 插件配置。

**模板 E：排错**
> 报错如下[贴完整堆栈]。相关代码[贴]。配置[贴]。请定位根因并给最小修改，解释为什么错。
> 不要直接重写整个类。

## 4. 编码规范（让 AI 遵守，写进每次提示）

- 分层严格：controller 不写业务，service 不直接拼 SQL，mapper 只访问本服务库。
- 不把 entity 直接返给前端，用 DTO/VO。
- 写接口幂等；外部调用(Feign)配降级；事务边界清晰。
- 统一用 mall-common 的 Result/BizException，不要每个服务自己造。
- 配置进 Nacos，不硬编码；敏感信息用环境变量/Secret。
- 每个 public service 方法配测试。

---

## 5. 每个功能的验收清单（你来打勾）

- [ ] 能讲清这段代码在做什么、为什么这么做（不能就让 AI 再解释）
- [ ] 编译通过、本地能跑通该接口（Knife4j/Postman 验证）
- [ ] 有单元测试且通过；涉及 DB/并发的有集成测试
- [ ] 异常路径有处理（库存不足、下游故障、重复请求）
- [ ] 用了统一 Result/异常/日志
- [ ] commit 信息清晰（一个功能一个/几个有意义的 commit）

---

## 6. 常见坑（提前告诉 AI 规避）

| 坑 | 正确做法 |
|---|---|
| 用过时的 Nacos bootstrap 配置 | 用 `spring.config.import`（Nacos 3.x/新 SCA） |
| 跨服务直接连别人的库 | 只能 Feign/MQ，每服务独立库 |
| 秒杀也套 Seata | 秒杀走 Redis+Lua+MQ 最终一致，别用全局事务 |
| 集成测试用 H2 假装 MySQL | 用 Testcontainers 起真 MySQL |
| 缓存只设固定过期 | 加随机抖动防雪崩；空值/布隆防穿透 |
| MQ 消费不幂等 | 去重表/唯一索引，重复消费不出错 |
| 把 AI 给的代码原样抄、不理解 | 每块都要能复述，否则面试必崩 |

---

## 7. 学习闭环建议

每完成一个难点，做三件事：① 让 AI 出 3 道该知识点的面试题考你；② 你写一段"如果面试官问我会怎么答"
（存进 README 的面试脚本区）；③ 跑一次测试/压测留截图。三个月下来，你不仅有项目，还有一套
背熟的、有数据支撑的、自己真懂的讲述——这才是简历"含金量"的来源。

---

## 8. 诚信底线

简历只写你真正实现并理解的模块。如果某块是 AI 写的你没吃透，要么花时间吃透，要么不写进简历。
面试官一个追问就能区分"做过"和"抄过"，后者的代价远大于多写一行项目经历的收益。
