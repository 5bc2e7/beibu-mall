## 1. 分布式事务：下单扣库存（Seata AT）

**问题**：下单要在 order 库插订单、在 inventory 库预占库存、（可能）在 promotion 扣券，跨 3 库 3 服务。
任一步失败必须整体回滚，否则订单/库存对不上。

**方案**：Seata **AT 模式**（自动补偿，无侵入）。order 作为发起方，方法上 `@GlobalTransactional`；
各分支服务的数据源用 Seata 代理（`DataSourceProxy`），并各建 `undo_log` 表。

```java
@GlobalTransactional(rollbackFor = Exception.class)
public String createOrder(OrderCreateDTO dto) {
    // 1) 校验商品&算价 (feign product)
    // 2) 预占库存 (feign inventory.lock) —— 失败抛异常 → 全局回滚
    // 3) 扣券 (可选)
    // 4) 插入本地订单
    return orderNo;
}
```

**取舍**：AT 模式靠全局锁保证写隔离，**高并发下全局锁是瓶颈**，所以秒杀链路不用 Seata（见第 6 节）。
TCC 更灵活但侵入大、要写三个方法，学习项目用 AT 足够，能讲清"为什么秒杀不用 Seata"即可。

**面试**：要能画出 TC/TM/RM 三角色、讲清 AT 两阶段（一阶段记 undo_log 直接提交本地事务，
二阶段全局提交则删 undo_log、全局回滚则用 undo_log 反向补偿）。

## 2. 库存超卖防护（普通商品）

**问题**：多请求同时扣同一 SKU，读到旧库存导致超卖。

**方案**：**数据库乐观锁**，把"判断 + 扣减"做成一条原子 SQL（见 03 第 3 节）：
```sql
UPDATE t_inventory SET available = available - #{qty}, locked = locked + #{qty}, version = version + 1
WHERE sku_id = #{skuId} AND available >= #{qty};
```
`affectedRows == 0` 即失败。配合 Seata 保证跨服务一致，配合库存流水可对账。

**为什么不用悲观锁**：`SELECT ... FOR UPDATE` 会锁行降低并发；乐观锁在冲突不极端时吞吐更好。
秒杀这种**极端高并发**才需要把扣减搬到 Redis（第 6 节）。

## 3. 分布式锁（Redisson）

**用途**：防止同一资源被并发重复操作，如"同一优惠券只领一次""定时任务多实例只跑一个"。

```java
RLock lock = redissonClient.getLock("lock:coupon:" + couponId + ":" + userId);
if (lock.tryLock(2, 10, TimeUnit.SECONDS)) {     // 等待2s，持有最多10s(看门狗自动续期)
    try { /* 领券逻辑 */ } finally { lock.unlock(); }
}
```
**要点**：必须设过期防死锁；Redisson 看门狗自动续期；解锁要判断是不是自己加的（Redisson 内部已处理）。
**面试**：能对比"SETNX+过期"自己实现的坑（非原子、误删别人锁、不可重入），说明为什么用 Redisson。

## 4. 缓存三剑客（穿透 / 击穿 / 雪崩）

商品详情、SKU 这类**读多写少**的数据进 Redis。要处理三个经典问题：

| 问题 | 现象 | 方案 |
|---|---|---|
| 穿透 | 查不存在的 key，每次打到 DB | 缓存空值(短 TTL) + 布隆过滤器拦截非法 ID |
| 击穿 | 热点 key 过期瞬间大量请求压 DB | 互斥锁(Redisson)重建缓存 / 逻辑过期 |
| 雪崩 | 大量 key 同时过期 | 过期时间加**随机抖动**；多级缓存；限流兜底 |

缓存与 DB 一致性用 **Cache-Aside + 延迟双删**（更新时先删缓存→改库→延迟再删一次），或商品变更
发 MQ 异步刷新缓存与 ES。

## 5. 限流 / 熔断 / 降级（Sentinel）

- **网关层限流**：对秒杀接口、下单接口设 QPS 阈值，超出快速失败（返回"排队中/稍后再试"）。
- **服务层熔断**：Feign 调用配 Sentinel/熔断兜底方法，下游(如搜索)挂了不拖垮主链路。
- **热点参数限流**：对秒杀某个 activityId 单独限流，防单点打爆。
- **降级**：非核心功能(推荐、评价)异常时返回兜底数据，保证浏览/下单可用。

```java
@SentinelResource(value = "createOrder", blockHandler = "createOrderBlock",
                  fallback = "createOrderFallback")
public String createOrder(...) { ... }
```
**面试**：讲清"限流(主动控流量)、熔断(下游故障时断开)、降级(返回兜底)"三者区别，以及 Sentinel
的滑动窗口与令牌桶/漏桶思想。

## 6. 秒杀（项目最硬的一块）⭐

时令海鲜限时抢购。目标：**高并发不超卖、不击穿 DB、防刷、最终一致**。

**整体链路**：
```
活动开始前：库存预热 → 写入 Redis  seckill:stock:{activityId}
请求进来：
  ① 网关/Sentinel 限流(QPS + 热点参数限流)
  ② 校验活动时间、用户是否已购(Redis Set 去重)
  ③ Redis 执行 Lua 脚本：原子判断库存>0 则扣减并标记已购  ← 防超卖、防重复
  ④ 扣成功 → 发 RocketMQ 消息(activityId,userId)，立即返回"排队中" + token
  ⑤ seckill 消费者异步：写 t_seckill_order(唯一索引 user+activity 兜底) → 创建正式订单
  ⑥ 前端用 token 轮询结果接口拿订单号
```

**Lua 原子扣减（防超卖核心）**：
```lua
-- KEYS[1]=stock key, KEYS[2]=bought set, ARGV[1]=userId
if redis.call('sismember', KEYS[2], ARGV[1]) == 1 then return -1 end  -- 已购
local stock = tonumber(redis.call('get', KEYS[1]))
if stock <= 0 then return 0 end                                        -- 售罄
redis.call('decr', KEYS[1])
redis.call('sadd', KEYS[2], ARGV[1])
return 1                                                               -- 成功
```

**为什么这样设计（取舍，必背）**：
- 扣减放 Redis + Lua：单线程原子，承载远高于 DB 的 QPS，**杜绝超卖**。
- 异步落库 + MQ：把"下单峰值"削平（**削峰**），DB 按自己节奏消费。
- 不用 Seata：全局锁会让 TPS 掉到几百；这里换成**最终一致**（Redis 已扣即视为成功，DB 异步跟上，
  失败则回补 Redis 库存）。
- DB 唯一索引 `(user_id, activity_id)`：兜底防重复下单（MQ 重复消费也不会出双单）。

**压测**：用 JMeter 对 `/buy` 接口压，记录优化前(直连 DB) vs 优化后(Redis+MQ) 的 TPS / P99 / 超卖数量，
这组对比数据是简历里最有说服力的一行。

## 7. MQ 的三大用途（RocketMQ）

| 场景 | 类型 | 说明 |
|---|---|---|
| 订单超时取消 | 延时消息 | 下单发 30min 延时消息，到点检查未支付则取消+释放库存 |
| 支付成功通知订单 | 普通消息 | 支付回调后发消息，订单服务消费转状态 |
| 削峰(秒杀落库) | 普通消息 | 见第 6 节 |
| 商品变更同步 ES | 普通消息 | 商品改动发消息，搜索服务更新索引 |

**消息可靠性**：生产端用**事务消息/本地消息表**保证"业务成功才发消息"；消费端**幂等**(去重表/唯一索引)
保证"重复消费不出错"；消费失败重试 + 死信队列。

---

## 8. 幂等性（贯穿全项目）

- 支付回调：`t_payment` 对 `order_no` 唯一索引，重复回调直接命中返回成功。
- 下单防重复提交：前端拿 token，后端 Redis 校验一次性 token。
- MQ 消费：消费前查去重表/Redis，处理过则跳过。

## 面试问答速记（本章相关）

- **Seata AT 二阶段做了什么？** → 一阶段记 undo_log 并提交本地；二阶段提交删 undo_log，回滚用 undo_log 反向 SQL。
- **秒杀为什么不用分布式事务？** → 全局锁吞吐低；改 Redis 扣减保证不超卖 + MQ 异步落库做最终一致 + DB 唯一索引兜底。
- **缓存和数据库怎么保证一致？** → Cache-Aside + 延迟双删，或变更发 MQ 异步刷新；强调"无法做到强一致，做到最终一致并可接受"。
- **Redis 怎么防超卖？** → Lua 原子脚本把"判断+扣减+去重"合成一步，单线程执行无并发问题。
- **限流和熔断区别？** → 限流主动控入口流量；熔断是下游故障时快速失败避免雪崩。
